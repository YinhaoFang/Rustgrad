### 第 2 页：问题陈述

- **核心问题**：深度学习框架内部是如何工作的？
- 大多数课程调用 `model.fit()` 即可训练，但内部张量运算、自动求导、梯度下
  降的具体机制对学生而言是黑盒。

## 二、演讲逐页讲稿

### 开幕（约 30 秒）

> 大家好，我的课程项目是 RustGrad——一个用 Rust 语言从零实现的深度学习教学
> 框架。这个项目的目标是：让大家看到当你调用 `model.fit()` 时，框架内部到底
> 发生了什么。

### 第 2-3 页：为什么做 & 怎么做（约 1 分钟）

> 深度学习框架对大多数同学来说是一个黑盒。我们这个项目选择的方式是：不调用
> 任何外部库，全部手写，从而暴露框架的每一层内部机制。
>
> 项目采用分层架构。最底层是张量——所有数值计算的载体。往上是自动求导引擎
> ——负责计算梯度。再往上是神经网络层、损失函数和优化器。训练循环把所有组件
> 编排在一起。最上层是命令行工具和报告导出。

### 第 4 页：张量（约 40 秒）

> 张量就是带形状信息的多维数组。我们实现了一个简洁的 Tensor 类型——支持加
> 减乘除、广播、矩阵乘法、转置、reshape、求和求均值等操作。每个操作都带形
> 状校验，出错时返回明确的错误类型而不是 panic。这个模块约 1150 行代码。

### 第 5 页：自动求导（约 1.5 分钟，**核心**）

> 自动求导是深度学习框架的心脏。我们的设计有三个核心概念：NodeId 是图中节点
> 的地址，GraphNode 存储值、梯度、父节点和操作类型，ComputationGraph 是图的
> 容器。
>
> 反向传播时分四步：先做拓扑排序确定计算顺序，然后从输出端以全 1 梯度初始
> 化，逆序遍历图，对每个操作查表应用对应的梯度规则，最后把梯度累加到父节点。
>
> 一个关键设计是 `backward_with_grad`。我们并不在图中计算损失函数的梯度——
> 而是手工算出损失对预测值的导数作为"种子"，然后交给 autograd 引擎继续反
> 向传播到参数。这样做的好处是：手工梯度公式容易验证，而参数梯度的链式传播
> 仍然享受了 autograd 的通用性。
>
> 目前实现了 12 种梯度规则，覆盖了框架中所有操作。每个梯度规则都经过了双重
> 验证：与手算公式的解析对比，以及与有限差分数值梯度的比较。

### 第 6-7 页：网络层 & 优化器（约 1 分钟）

> 神经网络组件通过 Module trait 统一接口。Linear 层实现的是标准的 y = xW+b。
> Sequential 容器可以把任意层串起来。
>
> 优化器方面，实现了三种：SGD 是最基础的梯度下降；Momentum 引入速度累积来平
> 滑更新；Adam 进一步引入了一阶矩和二阶矩的自适应估计和偏差校正。

### 第 8 页：训练循环（约 40 秒）

> 四个训练示例展示了递进的复杂度。最简单的线性回归只用单层 + MSE。二分类加
> 入了 sigmoid 激活和 accuracy 指标。XOR MLP 引入了隐层和非线性激活函数。
> 螺旋分类还需要做特征映射——因为原始坐标线性不可分，我们用极坐标变换把问
> 题变得可解。

### 第 9-10 页：周边基础设施（约 30 秒）

> 模型序列化选择了纯文本格式——rank 和维度作标头，每行一个 f64。17 位有效数
> 字保证保存再载入后值不变。数据集方面支持 CSV 加载和确定性的洗牌拆分。
> CLI 支持三种输出格式和报告文件导出。

### 第 11 页：质量（约 30 秒）

> 292 个测试全部通过，Clip​​py 零警告。特别值得一提的是 autograd 的有限差分验
> 证——我们真的对每个元素加一个微小的扰动，然后比较数值梯度和自动求导的结果。

### 第 12-14 页：总结（约 40 秒）

> 总结一下，这个项目从零实现了深度学习框架的核心路径——张量运算、自动求导、
> 网络层、优化器、训练、序列化、CLI 和报告。代码约 4000 行，零外部依赖。整
> 体来说，它既是一个可用的训练工具，也是一份可阅读的教学材料。

## 三、预备专业提问与答案

### Q1：为什么选择 Rust 而不是 Python？

**参考答案**：

Rust 的强类型系统和所有权模型迫使开发者在编译期处理所有潜在错误——没有
`NoneType has no attribute` 这类运行时意外。对于教学框架而言，这意味着：
- 张量形状不匹配在调用处立即被发现（通过 `Result<T, RustGradError>`），而不
  是训练 10 分钟后才崩溃。
- 所有权语义天然适合管理梯度生命周期（谁拥有梯度？何时释放？）。
- 零成本抽象：trait 和泛型不引入运行时开销。

Python 的灵活性和生态优势在应用层无可替代，但作为"理解框架内部"的教学工
具，Rust 的严格约束反而是优势。

### Q2：你的 autograd 如何实现拓扑排序？复杂度是多少？

**参考答案**：

使用深度优先搜索（DFS）+ visited 集合。从输出节点递归访问父节点，全部依赖
访问完毕后才将当前节点加入结果列表。时间复杂度 O(V+E)，其中 V 是节点数，E
是边数。每个节点恰好入 visited 集合并被推入一次，shared parent 不会被重复
访问——这是通过 `HashSet::insert` 的返回值（首次插入返回 true）来判断的。

### Q3：为什么梯度存在 `GraphNode` 而不是 `Tensor` 中？

**参考答案**：

三个原因：
1. **关注点分离**：Tensor 是纯数值容器——它不"知道"自己是训练参数还是中间结
   果。梯度信息是计算图层面的概念，属于图的元数据。
2. **教学清晰性**：在代码和文档中，张量值、图节点、累积导数三个概念有明确的
   类型边界，便于分别讲解。
3. **优化器独立性**：当前设计中优化器通过 `GradientSet` 接收梯度，与 autograd
   引擎解耦。这使我们可以单独测试优化器的更新规则而不需要构建完整的计算图。

PyTorch 的设计（梯度存储在 tensor 上）更省内存且支持原地操作，但需要更复杂
的引用计数和生命周期管理——超出教学框架的范围。

### Q4：你的 Softmax 反向传播如何避免 O(n²) 的雅可比矩阵？

**参考答案**：

Softmax 的完整雅可比矩阵 J 大小为 n×n，其中 J[i][j] = s[i]*(δ[i][j] - s[j])。
直接构造需要 O(n²) 空间。但实际上我们只需要向量-雅可比积 vjp = upstream_grad · J。

公式化简为：`grad[i] = s[i] * (upstream[i] - Σ_j s[j] * upstream[j])`

只需要 O(n) 时间和空间：先算 dot = Σ s[j]*upstream[j]，然后逐元素计算
`s[i]*(upstream[i] - dot)`。这就是 `softmax_gradient_slice` 的实现。

对于矩阵输入（批处理），我们对每一行独立应用上述公式——因为 softmax 是逐行
归一化的，行与行之间的雅可比为零。

### Q5：`backward_with_grad` 的设计思路是什么？为什么不把 Loss 也放入图中？

**参考答案**：

把 Loss 放入计算图有两种方式：
- 方案 A：在图中实现 MSE/CE 的梯度规则。可行，但每个损失函数都需要一个对应
  的 `Operation` 变体。
- 方案 B（我们的选择）：手工计算损失对最终输出节点的梯度，作为种子传入
  `backward_with_grad`。

方案 B 有几个优势：
- 损失梯度的公式非常短（MSE: `2*(pred-t)/N`，CE: `(softmax-t)/N`），手工
  实现代码比通用梯度规则更简洁也更易在教学报告中解释。
- 参数梯度的链式传播仍然由 autograd 处理——激活函数、矩阵乘法、转置等复杂
  求导不受影响。
- 测试友好：可以独立验证"损失梯度公式"和"图传播路径"两者的正确性。

### Q6：CPU 后端未来如何接入 GPU？

**参考答案**：

当前架构留了三个扩展点：
1. `Backend` trait 已经定义了全部核心运算的接口——GPU 后端只需实现这些方法。
2. 下一步需要在 `Tensor` 上引入泛型参数 `Tensor<B: Backend>`，使张量的数据
   存储与后端绑定（CPU 用 `Vec<f64>`，GPU 用设备内存句柄）。
3. `CpuBackend` 的零成本委托模式可直接泛化：GPU 后端在 `add` 等方法中将数
   据上传、启动 kernel、下载结果。

挑战主要在于异步执行和内存管理——GPU 操作是异步的，需要引入 stream/event 等
同步原语。Rust 的 ownership 模型恰好适合管理 GPU 内存生命周期（RAII 释放设
备内存）。

### Q7：训练循环每 epoch 重建计算图，性能如何？为什么不复用？

**参考答案**：

对于当前的小规模数据集（线性回归 31 样本、XOR 4 样本、Spiral 36 样本），
图构建开销可以忽略不计——每个 epoch 的总耗时几乎全部花在张量运算上。

不复用图的原因：
1. 图节点的 value 字段在 `add_operation` 时被 move 进去——要"刷新"一个已存在
   的图，需要暴露 `&mut Tensor` 来原地更新值，这会破坏当前的不变式设计。
2. 每个 epoch 的参数值不同（optimizer 已更新），所以图的中间值必须重算——
   重建的成本就是重新执行前向计算，这和复用图后重算的计算量是一样的。
3. 教学简单性：每轮"新建图 → 前向 → 反向 → 取梯度 → 优化器更新 → 丢弃图"
   的循环在概念上非常清晰。

对于生产级框架（如 PyTorch），图复用（`retain_graph`）和惰性重算确实重要，
但这个项目是教学框架，简单性优先。

### Q8：为什么 MSE 的梯度是 `(2/N)*(pred-target)` 而不是 `(1/N)*(pred-target)`？

**参考答案**：

MSE 的定义是 `L = (1/N) * Σ(pred - target)²`。对 pred 求导：
```
dL/dpred = (1/N) * 2 * (pred - target) = (2/N) * (pred - target)
```

2 来自于平方函数的导数 d(x²)/dx = 2x。在实现中（`train_linear_regression`）
我们确实使用了 `scale = 2.0 / n`，这与公式一致。有些框架会去掉 2 因子而改用
`0.5 * MSE` 来简化梯度，我们没有这样做——保持和标准定义一致。

### Q9：如何处理数值稳定性问题？

**参考答案**：

项目中有三个地方专门处理了数值稳定性：
1. **Sigmoid**（分段实现）：对于 x ≥ 0 用 `1/(1+exp(-x))`；对于 x < 0 用
   `exp(x)/(1+exp(x))`。避免了大正数 exp 溢出和大负数 exp 趋零的精度损失。
2. **Softmax**（减去最大值）：计算 `exp(x - max(x))` 而不是 `exp(x)`，保证
   指数运算前值域控制在 (-∞, 0] 范围内，避免 exp(1000) = inf。
3. **CrossEntropyLoss**（log-sum-exp + epsilon clip）：使用
   `max_logit + log(Σ exp(x - max))` 避免 log(0)，然后对概率值做
   `max(p, 1e-12)` 裁剪防止 log(0)。

### Q10：如果有更多时间，你会优先改进什么？

**参考答案**：

按优先级排列：
1. **Mini-batch 训练**：当前数据迭代器和 batch 接口已经就绪，训练循环层面的
   改造量不大，但能显著扩展可用的数据规模。
2. **`--load-model` CLI 选项**：训练前从 checkpoint 恢复权重继续训练或做推理。
3. **Sequential 泛化序列化**：遍历所有 layer 的 `parameters()` 自动保存/加载，
   支持任意深度和组合的网络。
4. **GPU 后端**：将 `CpuBackend` 的模式复制到 CUDA/wgpu 后端，同时改造
   `Tensor` 支持泛型后端。
5. **更多层类型**：Dropout、BatchNorm、简单的 Conv2D——每种约 100-200 行，但
   能大幅扩展模型表达能力。

### Q11：Adam 的偏差校正是怎么做的？为什么需要？

**参考答案**：

Adam 维护两个指数移动平均：一阶矩 m 和二阶矩 v。在 t=0 时 m 和 v 初始化为
0，所以前几步它们会偏向 0。偏差校正因子是 `1 - βᵗ`，其中 β 是对应的衰减
系数。

在代码中：
```rust
let beta1_correction = 1.0 - self.beta1.powf(self.timestep as f64);
let beta2_correction = 1.0 - self.beta2.powf(self.timestep as f64);
...
let first_hat = *first / beta1_correction;
let second_hat = *second / beta2_correction;
```

随着 timestep 增长，`βᵗ → 0`，校正因子 `→ 1`，所以校正效果在早期最强，逐
渐减弱。这是 Adam 论文的标准算法。

### Q12：项目的测试策略是什么？如何保证 autograd 的正确性？

**参考答案**：

三层验证：
1. **解析验证**：每个梯度规则都写成单元测试，autograd 输出与手算公式逐值对
   比（如 `z = x*y + x` 时 dz/dx = y+1 = 5.0）。
2. **有限差分验证**：`check_gradient_numeric` 框架对每个叶节点元素施加 ±1e-6
   扰动，用中心差分 `(f(x+h) - f(x-h)) / 2h` 近似真实梯度，与 autograd 对比。
   容差设在 1e-5，对 f64 足够严格。五个新梯度规则都通过了这个检验。
3. **收敛验证**：训练测试检查 loss 随 epoch 下降且最终收敛到合理值（如线性
   回归 loss < 1e-6，XOR accuracy = 100%）。如果梯度有误，模型不可能收敛到
   正确参数。

### Q13：为什么不使用任何外部依赖？这不是重复造轮子吗？

**参考答案**：

这正是项目的教学目标。"造轮子"在这个上下文中不是缺点而是手段——只有亲手
实现矩阵乘法、反向传播、参数更新，才能真正理解框架内部的工作原理。工程
实践中当然应该使用成熟的库，但教学实验中"理解轮子如何转动"比"用轮子跑得
快"更有价值。

具体到 Rust 生态：`nalgebra` 可以做张量，`burn` 和 `candle` 是现成的
深度学习框架——但它们的高层 API 封装恰好隐藏了我们想要展示的细节。零依赖
也意味着评审老师不需要安装任何额外的 crate 就能 `cargo test` 跑通全部测试。

### Q14：你的 autograd 和 PyTorch 的 autograd 有什么异同？

**参考答案**：

**相同点**：
- 都是动态计算图（define-by-run），每次前向构建图。
- 都使用反向模式自动求导（reverse-mode AD）。
- 梯度都在反向传播中累积到叶子节点。

**不同点**：
- PyTorch 的梯度存在 Tensor 上（`tensor.grad`），我们的梯度存在 GraphNode
  上。前者更省内存，后者概念更清晰。
- PyTorch 实现了完整的梯度规则覆盖（数百种操作），我们覆盖了教学中关键的
  12 种。
- PyTorch 支持原地操作和 `retain_graph`，我们每 epoch 重置图。
- PyTorch 的 C++ 后端做 JIT 编译和算子融合，我们是纯 Rust CPU 实现。

本质上，我们的 autograd 是 PyTorch 设计思想的"最小教学子集"——足够展示核心
原理，但远不到生产级的完整性和性能。
